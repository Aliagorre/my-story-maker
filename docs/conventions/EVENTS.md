# **StoryMaker V7 — Event Conventions**

---

## **1. Introduction**

Events are the primary communication mechanism between:

* the core
* core mods
* default mods
* external mods

The event system must be:

* **simple**
* **predictable**
* **strictly structured**
* **compatible with both sync and async execution**
* **generic enough** to support any type of mod

This document defines:

* the structure of an event
* official categories
* naming conventions
* required fields
* best practices

---

## **2. Event Structure**

An event is a Python dictionary with the following structure:

```python
{
  "name": "ENGINE_READY",
  "source": "core",
  "payload": {},
  "timestamp": 1234567890
}
```

### **2.1 Required Fields**

| Field       | Type   | Description                          |
| ----------- | ------ | ------------------------------------ |
| `name`      | string | Event name (UPPERCASE)               |
| `source`    | string | Mod name or `core`                   |
| `payload`   | dict   | Event data                           |
| `timestamp` | int    | UNIX timestamp generated by the core |

---

### **2.2 Rules**

* `payload` **must never be `None`**
  Always use a dictionary, even if empty.

* `source` must be:

  * `"core"` for engine events
  * `"mod_<name>"` for mods

---

## **3. Event Naming**

### **3.1 Official Format**

```id="e1"
CATEGORY_ACTION
```

* Always **UPPERCASE**
* Words separated by underscores

---

### **3.2 Official Categories**

| Category      | Description                      |
| ------------- | -------------------------------- |
| `ENGINE_*`    | Core lifecycle and system events |
| `MOD_*`       | Mod-related events               |
| `UI_*`        | User interface events            |
| `ADVENTURE_*` | Adventure-related events         |
| `SCRIPT_*`    | Script/DSL events                |
| `FS_*`        | File system events               |

---

### **3.3 Examples**

* `ENGINE_BOOT`
* `ENGINE_READY`
* `ENGINE_TICK`
* `MOD_ERROR`
* `MOD_LOADED`
* `UI_RENDER`
* `ADVENTURE_LOADED`
* `SCRIPT_EXECUTION_FAILED`
* `FS_FILE_READ`

---

## **4. Core Events**

The core emits the following events:

### **4.1 Lifecycle**

| Event             | Description                 |
| ----------------- | --------------------------- |
| `ENGINE_BOOT`     | Engine startup              |
| `ENGINE_INIT`     | Core structures initialized |
| `ENGINE_READY`    | All mods are ready          |
| `ENGINE_TICK`     | Main loop tick              |
| `ENGINE_SHUTDOWN` | Engine shutdown             |

---

### **4.2 Mod Management**

| Event                  | Description                   |
| ---------------------- | ----------------------------- |
| `MOD_DISCOVERED`       | A mod has been found          |
| `MOD_MANIFEST_ERROR`   | Invalid manifest              |
| `MOD_DEPENDENCY_ERROR` | Dependency resolution failure |
| `MOD_CONFLICT`         | Conflict detected             |
| `MOD_LOADED`           | `on_load` executed            |
| `MOD_INITIALIZED`      | `on_init` executed            |
| `MOD_ERROR`            | A mod raised an error         |

---

### **4.3 Core Errors**

| Event                | Description                       |
| -------------------- | --------------------------------- |
| `ENGINE_ERROR`       | Internal core error               |
| `ENGINE_FATAL_ERROR` | Critical error requiring shutdown |

---

## **5. Mod-Defined Events**

Mods may emit their own events, provided they follow:

* the defined structure
* official categories
* naming conventions

Examples:

* `UI_BUTTON_CLICKED`
* `ADVENTURE_SCENE_CHANGED`
* `SCRIPT_VARIABLE_UPDATED`
* `FS_DIRECTORY_CREATED`

---

## **6. Best Practices**

### **6.1 Always Use Structured Payloads**

❌ Incorrect:

```python
payload = None
```

✔ Correct:

```python
payload = {}
```

✔ Preferred:

```python
payload = {
  "scene_id": "intro",
  "previous_scene": "menu"
}
```

---

### **6.2 Avoid Generic Events**

❌ `EVENT_OCCURRED`
❌ `SOMETHING_HAPPENED`
❌ `UPDATE`

✔ `ADVENTURE_SCENE_CHANGED`
✔ `UI_WINDOW_OPENED`
✔ `MOD_ERROR`

---

### **6.3 Do Not Use Events for Direct Calls**

Events are not a replacement for:

* services
* stable APIs
* direct method calls

Events are for **notification**, not **invocation**.

---

### **6.4 Avoid Event Overuse**

A mod should not:

* spam `ENGINE_TICK`
* emit unnecessary events
* use events as an RPC mechanism

---

## **7. Summary**

| Element    | Rule                                     |
| ---------- | ---------------------------------------- |
| Format     | `CATEGORY_ACTION`                        |
| Fields     | `name`, `source`, `payload`, `timestamp` |
| Payload    | Always a dictionary                      |
| Categories | ENGINE, MOD, UI, ADVENTURE, SCRIPT, FS   |
| Usage      | Notification, not RPC                    |
| Source     | `"core"` or `"mod_<name>"`               |
