# StoryMaker V7 — EventBus Specification

## 1. Introduction

The EventBus is the central communication mechanism between:

- the core,  
- core mods (`mods_core`),  
- default mods,  
- external mods.

It enables:

- emitting events,  
- subscribing to events,  
- propagating information without direct dependencies,  
- structuring engine logic around an event-driven flow.

The EventBus must be:

- **simple** to use,  
- **robust** against errors,  
- **performant**,  
- **compatible with both sync and async**,  
- **completely independent** from business logic.

---

## 2. Objectives of the Event System

- Decouple mods from each other.  
- Allow the core to broadcast lifecycle stages.  
- Enable mods to react to events without knowing other mods.  
- Provide a stable and extensible system.  
- Ensure that errors in handlers never stop the engine.

---

## 3. Event Structure

An event is a structured Python dictionary:

```python
{
  "name": "ENGINE_READY",
  "source": "core",
  "payload": {},
  "timestamp": 1234567890
}
```

### 3.1 Required Fields

| Field       | Type   | Description                          |
| ----------- | ------ | ------------------------------------ |
| `name`      | string | Event name (UPPERCASE)               |
| `source`    | string | `"core"` or `"mod_<name>"`           |
| `payload`   | dict   | Associated data                      |
| `timestamp` | int    | UNIX timestamp generated by the core |

### 3.2 Rules

* `payload` must always be a dict (never `None`).
* `source` is automatically added by the core.
* `timestamp` is automatically added by the core.

---

## 4. EventBus API

The EventBus is **not directly exposed** to mods.
Mods interact with it through the core API:

* `core.emit()`
* `core.subscribe()`

For internal documentation purposes, the full API is described below.

---

### 4.1 `emit(event: dict)`

Emits an event into the system.

#### Internal Signature

```python
emit(event: dict)
```

#### Rules

* The EventBus validates the event structure.
* All subscribed handlers are called.
* Errors in a handler:

  * are caught,
  * are logged,
  * do not stop propagation.

---

### 4.2 `subscribe(event_name: str, callback: Callable)`

Subscribes a handler to an event.

#### Internal Signature

```python
subscribe(event_name: str, callback: Callable)
```

#### Rules

* `callback(event)` must accept exactly one argument.
* A mod can subscribe to any number of events.
* Subscriptions are stored internally:

```
subscriptions[event_name] = [callback1, callback2, ...]
```

---

### 4.3 `unsubscribe(event_name: str, callback: Callable)` *(optional)*

Unsubscribes a handler.

#### Internal Signature

```python
unsubscribe(event_name: str, callback: Callable)
```

---

## 5. Execution Order

### 5.1 Handler Order

Handlers are executed:

1. in order of subscription,
2. synchronous handlers first,
3. then asynchronous handlers (if async mode is enabled).

### 5.2 Non-blocking Requirement

* Handlers must not block execution.
* Long operations must use threads or async tasks.
* The core does not manage mod threading.

---

## 6. Error Handling

### 6.1 Handler Errors

If a handler raises an exception:

* the error is caught,
* a log entry is created,
* a `MOD_ERROR` event is emitted,
* propagation continues.

### 6.2 EventBus Internal Errors

If an internal error occurs:

* an `ENGINE_ERROR` event is emitted,
* a detailed log is produced,
* the engine continues if possible.

---

## 7. Performance

The EventBus must be:

* **O(1)** for subscription,
* **O(n)** for emission (n = number of handlers),
* capable of handling hundreds of events per second,
* suitable for dozens of mods.

No premature optimization is required, but the design must remain simple.

---

## 8. Sync / Async Compatibility

The EventBus must support two modes:

### 8.1 Synchronous Mode (default)

* All handlers are executed immediately.
* Ensures simplicity and predictability.

### 8.2 Asynchronous Mode (optional)

* Async handlers are awaited.
* Sync handlers run normally.
* The engine may use an `asyncio` loop.

### 8.3 Rules

* Mods may use sync or async freely.
* The core never enforces async usage.

---

## 9. Core Reserved Events

The EventBus must support the following core events:

* `ENGINE_BOOT`
* `ENGINE_INIT`
* `ENGINE_READY`
* `ENGINE_TICK`
* `ENGINE_SHUTDOWN`
* `ENGINE_ERROR`
* `ENGINE_FATAL_ERROR`
* `MOD_DISCOVERED`
* `MOD_LOADED`
* `MOD_INITIALIZED`
* `MOD_ERROR`
* `MOD_MANIFEST_ERROR`
* `MOD_DEPENDENCY_ERROR`
* `MOD_CONFLICT`

These events are defined in `EVENTS.md`.

| `subscribe`   | Subscribe to an event      |
| `unsubscribe` | Unsubscribe                |
| Sync/Async    | Fully supported            |
| Errors        | Caught, never blocking     |
| Structure     | Simple, stable, performant |
